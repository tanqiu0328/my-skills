# 编写嵌入式 C 注释

只提交能够从代码和项目配置中证实的注释。默认使用简体中文；保留
API 名称、协议字段、寄存器名、错误码、命令和代码标识符的原文。

## 工作流程

### 1. 建立事实边界

读取目标声明、实现、直接调用方、相关配置和同目录文件。确认项目的
注释格式、术语和语言约定。用户或项目明确指定其他语言时服从该约定，
否则使用简体中文。

不要根据函数名推断阻塞性、线程安全性、回调上下文、参数所有权或硬件
行为。沿调用链检查实现；无法证实的契约列为待确认问题，不写入注释。

完成条件：每项准备写入的事实都有代码、配置或现有项目文档作为依据。

### 2. 判断是否需要注释

只注释代码不能自行表达的意图和约束，包括：

- 设计原因、算法不变量、边界条件和错误恢复策略；
- 并发、时序、硬件、副作用、资源所有权和生命周期约束；
- 公共 API 的调用契约。

做一次覆盖扫描，不要求每行都有注释，但不能只停在公共 API 契约：

- 文件/模块：当文件包含协作状态、硬件职责、协议流程、RTOS 任务或
  编译配置分支时，用模块级注释说明功能边界、主要数据流、初始化/
  释放顺序和平台依赖；简单工具文件无需机械添加模块说明。
- 函数/代码块：状态机转换、关键算法阶段、错误恢复、资源交接、
  寄存器访问序列、临界区、延时/等待条件和超时策略，需要在块前或
  关键语句旁说明原因或约束。
- 嵌套语句：多层 `if`、`switch`、循环或条件编译中，给非显而易见
  的分支条件、循环不变量、退出条件、`break`/`continue`、有意贯穿
  和默认分支处理添加靠近代码的短注释；不要注释普通赋值或大括号。
- 条件编译：模块功能受宏选择影响时，说明配置意图、平台差异、互斥
  关系或退化行为。

按实体判断是否值得写：

- 宏：注释协议常量、硬件引脚/电平/时序、单位/范围、兼容别名和配置
  开关副作用；不注释头文件保护宏、明显布尔值、普通 `TAG`，或名称已
  充分表达的私有常量。
- 枚举：公共枚举类型说明类型含义；枚举值只在存在协议编码、状态机
  语义、兼容含义或非显而易见差异时说明。
- 结构体：不要因为 `typedef struct` 是公共类型就自动写 Doxygen 块。
  只有类型本身存在跨字段不变量、协议/硬件布局、所有权模型，或项目
  明确要求生成类型文档时，才写类型级注释。简单统计结构、字段名已
  清楚的状态快照和私有上下文结构默认不写类型级注释。
- 字段：优先把单位、范围、所有权、生命周期、是否可为 `NULL`、有效
  性依赖、缓存/对齐/`packed` 布局约束写在字段相邻位置；不写“保存
  xxx”“表示 xxx”这类复述字段名的注释。
- 函数指针和回调类型：说明调用上下文、阻塞限制、参数有效期，以及
  注册和注销责任。

不要复述语句表面行为。若名称或接口本身含糊，不要用注释掩盖；保持
有效代码不变，并在交付结果中报告问题。简单私有函数无需机械添加注释。

完成条件：每个目标文件已完成模块、公共 API、关键代码块和嵌套控制流
的覆盖扫描；每条新增注释都解释了代码本身未表达的意图、契约或误用
边界；宏、类型和字段没有被机械补齐。

### 3. 选择注释形式

严格按用途选择注释定界符：

- Doxygen 文档块只使用 `/** ... */`。
- Doxygen 以外的所有注释，包括普通说明、代码分区、私有函数和行内
  注释，只使用 `//`；多行注释的每行均以 `//` 开头，不使用
  `/* ... */`。
- 每条注释及每个 Doxygen 标签说明的最后一个非空白字符不得是中文
  句号 `。` 或英文句号 `.`。

按以下规则决定注释内容：

- 新文件：使用包含 `@file`、`@brief` 和 `@author 谈秋` 的
  Doxygen 文档块，`@brief` 说明模块功能边界而不是复述文件名。
- 已有文件缺少必要模块说明时，按项目风格补 `@file` 文档块、代码
  分区注释或文件顶部 `//` 说明；不要为了统一外观改写无关文件头。
- 公共函数：必须使用 Doxygen 文档块，包含准确的 `@brief`、每个
  `@param`、非 `void` 函数的 `@return`，以及必要的 `@note`。
- 公共类型、枚举和宏组：先判断是否真的需要注释。需要时按项目已有
  风格选择紧凑 Doxygen 或相邻 `//` 注释；字段级约束优先写在字段
  附近，不上移成复述类型名的 `@brief`。不要为了仿照外部示例新增
  `/*!< ... */`。
- 变量、结构体字段、枚举值和局部常量的短注释放在声明右侧；右侧注释
  超过 80 列或需要多句说明时，改放到声明上一行。
- 普通说明、代码分区、私有函数、关键语句和嵌套逻辑：仅记录非显而
  易见的原因或约束，注释位置应贴近读者会误判的代码。
- 回调、ISR、并发、异步、硬件或资源生命周期代码：读取
  [embedded-c-contract-comments.md](references/embedded-c-contract-comments.md)
  并应用所有相关条目。

参数文档应按事实注明空指针约束、范围、单位、缓冲区大小和所有权；
返回值文档应覆盖实际错误码及其含义，不得承诺实现没有保证的行为。

完成条件：所有公共 API 均有完整文档；需要说明的公共类型、枚举和
宏组已覆盖；每条注释的用途、定界符和末尾字符符合本节规则；所有条件
契约均已按需加载并覆盖。

### 4. 执行最小注释修改

按文件/模块、公共接口、关键代码块、嵌套控制流的顺序新增或更新必要
注释，删除过时、重复和注释掉的死代码。重写既有英文注释时保留技术
含义并改用简体中文。保持代码标识符、Doxygen 标签和第三方术语原文。

写成代码审查中会留下来的克制注释：

- 先说约束或原因，再说结论；不要写宣传式、教学式、总结式或说明书
  句子。
- 使用项目里已有的短语和命名，不为了显得正式而替换成生硬同义词。
- 普通 `//` 注释允许使用短语或半句，只要项目风格允许；不要为了凑
  完整句子写成“该函数用于...”“通过...来...”。
- 避免“用于”“负责”“确保”“相关”“进行”“实现对”“主要用于”
  “完成对”“提供一种”“提高可靠性”等空泛套话；能删不影响理解的词
  直接删。
- 避免“此函数”“该模块”“本接口”等模板开头，除非项目已有这种风格。
- 不写“提高可靠性”“保证稳定性”“方便维护”等无法从当前代码直接
  验证的价值判断。
- 不把代码标识符逐词翻译成中文解释；如果注释只是在翻译变量名、函数名
  或枚举值，删掉它。
- 同一文件中连续新增的注释不要复用同一句式。若多条注释都长得像模板，
  逐条压缩成贴近代码现场的事实。
- 一条注释只解释一个读代码时会卡住的问题；多个问题拆成相邻短句。

将注释控制在每行 80 列以内，并遵循相邻代码的缩进与注释布局。不要
修改任何有效代码、标识符、接口、预处理逻辑或程序行为。

完成条件：修改仅涉及注释文本或被注释掉的死代码，且没有冗余、模板化
或价值判断式注释；新增注释删掉后会让读者更容易误判约束、意图或契约。

### 5. 验证并交付

逐项核对修改后的注释与声明、实现、调用方和配置一致。检查：

- 公共 API 的参数、返回值和条件契约完整；
- 模块功能、关键语句、嵌套语句和条件编译的覆盖扫描已完成，缺注释处
  均有明确理由；
- 宏、类型和字段注释解释契约或约束，不是逐项机械补齐；
- 没有猜测、过时信息、表面复述或被注释掉的代码；
- 没有模板开头、空泛套话、教学腔、说明书腔或无法验证的价值判断；
- 注释定界符和末尾标点符合第 3 步；
- 变量、结构体字段、枚举值和局部常量的短注释位于声明右侧；
- 每行不超过 80 列，语言与项目约定一致；
- diff 未改变有效代码或程序行为。

若项目提供 Doxygen、编译或文档检查命令，运行与注释修改直接相关的
检查。报告已修改的注释、验证结果，以及未通过注释解决的命名、接口或
契约歧义。

完成条件：每个修改项均通过上述检查，所有未证实信息均已明确报告。
