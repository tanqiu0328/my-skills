# 编写嵌入式 C 注释

只提交能够从代码和项目配置中证实的注释。默认使用简体中文；保留
API 名称、协议字段、寄存器名、错误码、命令和代码标识符的原文。

## 工作流程

### 1. 建立事实边界

确定用户指定的目标文件或模块，并对目标 `.h`、`.c` 文件进行全文件
检查，而不是只查看已修改行。不要擅自把范围扩大到整个项目。

读取目标声明、实现、直接调用方、相关配置和同目录文件。确认项目的
注释格式、术语和语言约定。用户或项目明确指定其他语言时服从该约定，
否则使用简体中文。

不要根据函数名推断阻塞性、线程安全性、回调上下文、参数所有权或硬件
行为。沿调用链检查实现；无法证实的契约列为待确认问题，不写入注释。

完成条件：目标文件边界明确，每项准备写入的事实都有代码、配置或现有
项目文档作为依据。

### 2. 建立最低注释覆盖

对每个目标文件逐项满足以下最低覆盖要求：

| 位置 | 必须说明的内容 |
| --- | --- |
| `.h` 文件头 | 模块对外提供的能力和使用边界 |
| `.c` 文件头 | 模块内部职责、主要组成和实现策略 |
| `.h` 公共 API | 参数、返回值、副作用及适用的调用契约 |
| `.c` 公共函数实现 | 非简单实现的职责、主要流程或设计策略，不重复 `.h` 契约 |
| 非简单私有或 `static` 函数 | 函数职责、关键步骤及重要约束 |
| 模块功能分区 | 该段代码负责的功能阶段 |
| 关键控制逻辑 | 判断目的、状态不变量或不能直接从语句读出的约束 |

`.h` 和 `.c` 即使不是新文件，也必须检查并补齐准确的文件级 Doxygen。
已有准确文件头时保留或更新，不重复添加。`.h` 的公共 API 契约是唯一
来源；对应 `.c` 实现只补充实现说明。公共函数没有头文件声明时，在唯一
声明处记录契约，并报告接口组织问题。

在以下关键控制逻辑前使用普通注释：

- 两层及以上的嵌套条件或循环；
- 状态转换、模式切换和状态上报；
- 硬件操作顺序、时序敏感步骤和中断处理；
- 故障判断、错误恢复、降级和安全策略；
- 按键、泵控制、显示、周期任务等独立功能阶段。

简单 getter/setter、直接赋值、简单包装函数、日志语句和显而易见的
空值判断无需注释，除非它们承担额外契约或硬件约束。

说明模块功能、函数职责和关键流程，但不要逐行翻译代码。若名称或接口
本身含糊，不要用注释掩盖；保持有效代码不变，并在交付结果中报告问题。

完成条件：每个目标文件都有模块说明；每个非简单函数、功能分区和关键
控制逻辑均已注释，或符合上述免注释条件。

### 3. 选择注释形式

严格按用途选择注释定界符：

- Doxygen 文档块只使用 `/** ... */`。
- 普通说明、代码分区、函数实现说明和行内注释只使用 `/* ... */`，
  不使用 `/** ... */` 或 `// ...`。
- 每条注释及每个 Doxygen 标签说明的最后一个非空白字符不得是中文
  句号 `。` 或英文句号 `.`。

文件级 Doxygen 必须包含 `@file` 和准确的 `@brief`；新文件还应按项目
约定写入 `@author 谈秋`。公共 API 的 Doxygen 必须包含准确的
`@brief`、每个 `@param`、非 `void` 函数的 `@return`，以及必要的
`@note`。

普通说明、代码分区、函数实现说明和行内注释应概括功能、职责、流程或
约束，不复述参数和返回值契约。
- 回调、ISR、并发、异步、硬件或资源生命周期代码：读取
  [embedded-c-contract-comments.md](references/embedded-c-contract-comments.md)
  并应用所有相关条目。

参数文档应按事实注明空指针约束、范围、单位、缓冲区大小和所有权；
返回值文档应覆盖实际错误码及其含义，不得承诺实现没有保证的行为。

完成条件：文件和公共 API 均有完整 Doxygen；普通注释覆盖第 2 步要求
的位置；每条注释的用途、定界符和末尾字符符合本节规则；所有条件契约
均已按需加载并覆盖。

### 4. 执行最小注释修改

新增或更新必要注释，删除过时、重复和注释掉的死代码。重写既有英文
注释时保留技术含义并改用简体中文。保持代码标识符、Doxygen 标签和
第三方术语原文。

将注释控制在每行 80 列以内，并遵循相邻代码的缩进与注释布局。不要
修改任何有效代码、标识符、接口、预处理逻辑或程序行为。

完成条件：修改仅涉及注释文本或被注释掉的死代码，且没有冗余注释。

### 5. 验证并交付

逐项核对修改后的注释与声明、实现、调用方和配置一致。检查：

- 公共 API 的参数、返回值和条件契约完整；
- 每个目标文件、非简单函数、功能分区和关键控制逻辑达到最低覆盖；
- `.h` 契约与 `.c` 实现说明没有重复维护同一信息；
- 没有猜测、过时信息、表面复述或被注释掉的代码；
- 注释定界符和末尾标点符合第 3 步；
- 每行不超过 80 列，语言与项目约定一致；
- diff 未改变有效代码或程序行为。

若项目提供 Doxygen、编译或文档检查命令，运行与注释修改直接相关的
检查。报告已修改的注释、验证结果，以及未通过注释解决的命名、接口或
契约歧义。

完成条件：每个修改项均通过上述检查，所有未证实信息均已明确报告。
