# 变更请求管理

## 执行步骤

1. 完整读取 [ASPICE 文档契约](references/document-contract.md)、[过程参考](references/process.md) 和 [文档模板](assets/template.md)
   - 完成标准：已明确变更登记、影响分析、授权、实施和关闭的证据边界
2. 识别变更来源和实际影响范围
   - 先读取变更请求和直接关联工作产品，再按影响检查问题、需求、设计、代码、测试、配置和计划
   - 完成标准：变更动机、当前状态和可能受影响的工作产品已确认
3. 唯一标识并记录变更请求
   - 记录发起者、原因、范围、目标版本、状态和预期收益
   - 完成标准：变更边界明确，相关方对要改变什么有共同理解
4. 分析和评估影响
   - 识别受影响工作产品、其他变更依赖、资源、进度、风险和验证需求
   - 完成标准：实际受影响的工作产品均有处置，未受影响类别可用一条理由排除
5. 在实施前记录优先级和批准结论
   - 记录实际决策人、日期、条件和目标发布范围；小团队中同一人可兼任发起人与批准人，但必须留下明确决策
   - 完成标准：实施授权可追溯，未批准变更没有进入实现状态
6. 跟踪实施并确认结果
   - 建立变更与问题、需求、设计、代码、测试和配置项的双向追溯
   - 连接实现、评审和按影响选择的验证证据，由实际受影响方确认后关闭
   - 完成标准：每项已实施变更都有验证结果，未完成项有责任人和下一行动
7. 创建或更新 `change-log.md`，执行交付检查并报告路径、影响、决定和状态
   - 完成标准：变更从原因、影响、决策、实施到验证形成与风险相称的闭环
