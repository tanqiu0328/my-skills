# 嵌入式问题诊断

用一个能稳定判定症状的 **tight loop** 驱动诊断。问题管理和证据归档由 SUP.9 负责；本 Skill 只负责找到并验证原因

## 1. 建立反馈闭环

读取复现步骤、日志、版本、目标硬件和相关代码。优先建立能够由代理重复执行的单一判定入口：

- 主机单元或集成测试
- 可重复的串口、CAN、网络或文件输入回放
- 模拟器、QEMU、HIL 或板级脚本
- core dump、HardFault 栈帧、trace 或固定采样窗口
- 性能问题的基线测量命令

需要板卡、示波器或人工操作时，把动作、采集项和通过条件写成最短 HIL 步骤，并使用用户提供的结果继续

本步骤完成于已经实际运行或明确交接一个能命中用户原始症状的判定入口，并记录命令、环境、版本和输出；无法建立时，列出已尝试方法和缺少的最小证据后停止猜测

<HARD-GATE>
反馈入口尚未实际运行，且用户还不能执行已定义的 HIL 步骤时，本轮只交付反馈入口、采集字段和最小证据请求，然后结束。不要排列根因概率、给出修复代码或把诊断实验写成最终修复
</HARD-GATE>

## 2. 复现并最小化

重复运行反馈入口，确认它捕获的是用户描述的同一症状。逐项缩减输入、配置、任务、外设和调用链，每次缩减后重新运行

非确定性问题以提高复现率为目标，固定随机源、时间、负载和调度条件，记录样本数与失败率

本步骤完成于每个剩余条件都对复现必要，或已把不能继续缩减的硬件/时序条件写成明确边界

## 3. 排列可证伪假设

提出三至五个按证据排序的假设。每个假设写明：如果它成立，哪项观测会出现；哪项实验可以推翻它

涉及 MCU 异常、DMA/Cache、RTOS 并发、内存破坏或实时性时，完整读取 [嵌入式诊断探针](references/embedded-probes.md)，只选择能区分当前假设的探针

本步骤完成于每个假设都有不同的可观察预测和最小验证动作

## 4. 单变量验证

一次只验证一个假设。优先使用断点、watchpoint、trace、静态分析和现有测量点；新增临时日志时使用唯一前缀 `[DEBUG-<id>]`

性能问题先测量和二分，再修改代码。需要改动受控工作产品时，遵循项目变更规则；未获授权时只增加可逆诊断手段或报告建议

本步骤完成于一个假设被证据支持且其余高优先级假设已被推翻，或者所有假设均被推翻并回到步骤 3

## 5. 修复并锁定回归

在能覆盖真实故障链的 seam 上先保存失败证据或回归测试，再实施最小修复。运行最小复现和原始反馈入口，确认二者均恢复

没有合适测试 seam 时明确记录架构限制，不用浅层测试制造虚假信心

本步骤完成于原始症状不再出现、回归证据通过、受影响范围已验证，并且结论能从现象追溯到原因和修复

## 6. 清理与交接

删除全部 `[DEBUG-<id>]` 临时插桩和一次性产物，保留有长期价值的回归测试。报告症状、根因、证据、修复、回归结果和仍需板级确认的门槛

需要正式问题闭环时，将这些结果交给 `$aspice-resolve-problems`；暴露出架构 seam 问题时，在修复完成后建议 `$improve-codebase-architecture`

本步骤完成于仓库无遗留临时插桩，且后续人员可仅依赖交接内容复核结论
