# 嵌入式 C 风险检查表

这是变更风险审查，不是功能安全认证。先执行基础检查，再按 diff 触发风险分支；每个结论连接到命令、代码位置、工具结果或明确的人工判断

## 1. 基础检查

每次有效代码修改均检查：

- [ ] 使用项目真实命令完成目标构建，或记录无法构建的具体原因
- [ ] 新增编译警告为零；已有警告与本次改动分开记录
- [ ] 函数声明、定义、调用和配置保持一致
- [ ] 返回值、错误码和失败路径符合调用方契约
- [ ] 外部输入在首次信任前完成范围、长度、格式和单位校验
- [ ] 数组索引、长度计算、类型转换和循环边界可证明有效
- [ ] 修改范围内没有失效注释、临时调试代码和无依据的行为扩展

完成标准：每个修改 hunk 都至少连接到一项构建、分析、测试或可审查证据

## 2. 内存与数值分支

修改指针、缓冲区、结构体布局、分配、序列化或数值计算时检查：

- [ ] 缓冲区容量与有效长度分开表达，复制前验证源长度和目标容量
- [ ] 字符串复制保证终止符和截断语义，不把 `strncpy` 本身视为安全保证
- [ ] 分配器族、所有权、释放责任和错误路径清理一致
- [ ] 安全关键运行路径使用的动态分配具有项目依据、失败策略和碎片风险判断
- [ ] 释放后没有继续访问；置空只保护当前所有者，不用它替代别名分析
- [ ] 整数提升、符号转换、截断、溢出、移位和单位换算均在有效范围内
- [ ] 栈上对象与调用深度符合实际任务栈测量；大对象优先静态、池或项目认可的所有权方案

完成标准：每个内存对象和关键数值都有明确的范围、所有者和失效行为

## 3. 并发与 ISR 分支

修改任务、ISR、回调、DMA 完成路径或共享状态时检查：

- [ ] 列出生产者、消费者、执行上下文、优先级和对象所有者
- [ ] 共享访问使用与平台一致的原子操作、临界区、锁或消息传递
- [ ] `volatile` 只承担可观察访问，不被当作原子性、互斥或内存顺序保证
- [ ] ISR 只调用该平台允许的 API，执行时间有界且不进行阻塞或运行时分配
- [ ] 锁顺序、错误路径解锁、优先级反转和回调重入均有处理
- [ ] 无锁环形缓冲区明确 SPSC/MPSC 模型、索引原子性、回绕和发布顺序

完成标准：每项共享状态都能从所有访问上下文追到唯一同步策略

## 4. 硬件、DMA 与实时性分支

修改 HAL、寄存器、外设、DMA、Cache、时钟或时序时检查：

- [ ] HAL 调用的同步/异步、超时、回调上下文和错误语义已从实现或文档确认
- [ ] 时钟、引脚复用、访问宽度、读改写语义、中断优先级和错误标志与硬件一致
- [ ] DMA 缓冲区位于可访问内存区，生命周期覆盖传输，地址和长度满足硬件约束
- [ ] Cache clean/invalidate 使用正确方向、cache-line 对齐范围和传输前后顺序
- [ ] 等待、轮询和临界路径有明确上界，不破坏任务截止时间和看门狗策略
- [ ] 看门狗只由健康策略授权喂入，任务或错误路径不能无条件掩盖失活

完成标准：软件动作能够追到硬件可观察结果，未验证的平台事实已成为明确阶段门槛

## 5. 生命周期与影响分支

修改接口、状态、枚举、资源或初始化顺序时检查：

- [ ] 初始化、启动、停止、取消和清理状态转换完整
- [ ] 句柄、锁、定时器、线程、回调和外设资源在所有退出路径上按所有权释放
- [ ] 结构体、枚举、宏和公共接口变化已覆盖全部调用者与实现
- [ ] 启动顺序、升级兼容、持久化格式和失败恢复没有被意外改变
- [ ] 回归集覆盖正常路径、边界和本次新增的最高风险失败路径

完成标准：所有直接和间接受影响对象均已验证或记录下一行动

## 6. 交付记录

报告：

- 实际执行的构建、检查和测试及其结果
- 发现并修复的问题
- 未执行分支及“不适用”理由
- 需要目标硬件、长时间运行或人工设备补充的门槛
- 对内存、并发、实时性和硬件行为作出的假设

只有证据覆盖的范围可以标记为通过；清单通过不等于满足 MISRA、ISO 26262、IEC 61508 或其他认证要求
