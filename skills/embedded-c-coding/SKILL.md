# 嵌入式 C 编码

把项目现有规范、目标平台事实和可验证安全性作为约束。通用原则只补足项目未定义的部分，不用设计模式替代工程判断

## 1. 建立代码画像

先读取范围内代码、相邻模块、构建方式和项目规范，确认：

- 裸机、轮询、事件驱动或 RTOS 执行模型
- ISR、任务、DMA、回调和共享数据上下文
- 动态内存、栈、Cache、MPU、看门狗和错误处理策略
- 项目已有接口、命名、头文件、格式和测试方式

检测到 `platform_device`、`INIT_*_EXPORT`、活动对象或项目五层目录时，才完整读取 [架构与平台约定](references/architecture.md)

本步骤完成于变更边界、执行上下文、适用项目规则和不能假设的平台事实均已明确

## 2. 选择最小参考集

只加载当前分支需要的 reference：

| 变更分支 | 必须读取 |
|---|---|
| 公共接口、模块 seam、状态机或可替换实现 | [设计模式](references/design-patterns.md) |
| 函数职责、错误契约、命名或防御性检查 | [Clean Code](references/clean-code.md) |
| 动态内存、缓冲区、共享状态或可重入性 | [内存与并发安全](references/memory-safety.md) |
| HAL、寄存器、DMA、Cache、ISR 或外设时序 | [硬件交互](references/hardware-interaction.md) |
| 项目没有更具体的格式和类型规则 | [代码风格](references/code-style.md) |
| 任何有效代码修改或代码审查 | [风险检查表](references/safety-checklist.md) |

项目规范与 reference 冲突时，优先遵循已经确认且安全的项目规范；冲突会影响内存、并发、实时性或硬件正确性时，显式报告并请求决定

本步骤完成于每份已加载 reference 都对应当前变更分支，没有因“不确定”加载整套手册

## 3. 设计最小改动

保持项目已有架构和接口风格。只有行为确实变化、存在第二个实现或测试 seam 明确受益时才增加抽象

- 明确输入范围、单位、所有权、生命周期和错误契约
- 明确函数可重入性、线程安全性和回调上下文
- 硬件访问追到实际 HAL、寄存器、时钟和中断行为
- 事件驱动调用链保持有界；阻塞操作使用项目已有的异步机制
- 安全关键路径优先使用静态分配、固定容量或项目认可的内存池；运行时堆分配必须有项目依据

本步骤完成于改动能用现有 seam 实现，新增复杂度有真实需求，关键失败路径具有确定行为

## 4. 实现并局部验证

按项目风格修改代码，控制 diff 范围。先运行最快且能覆盖改动的构建、静态检查或测试；硬件条件缺失时保存可在目标环境执行的最小验证步骤

注释默认使用简体中文并解释意图、上下文或约束；项目有明确语言规范时保持项目一致。代码标识符、寄存器名、API 和错误码保持原文

本步骤完成于代码能够构建或已明确局部受阻原因，新增行为具有直接验证证据

## 5. 执行风险审查

完整读取并执行 [风险检查表](references/safety-checklist.md) 中与本次 diff 匹配的分支。每次后续修复都重新运行受影响分支和基础检查，不重复无关平台检查

对未执行项记录“不适用”及理由；不能实际验证的硬件、时序和资源假设记录为阶段门槛，不宣称通过

本步骤完成于每个修改点都被编译/分析/测试或人工证据覆盖，所有高风险项已解决或明确交接

## 6. 交付

报告修改文件、行为变化、验证命令与结果、内存/并发/实时性判断、硬件假设和剩余门槛。代码审查任务只报告发现，不修改有效代码，除非用户同时授权修复

本 Skill 完成于变更、风险与证据能够逐项对应，且没有把检查清单完成误写成安全认证结论
