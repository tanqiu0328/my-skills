# ASPICE 文档契约

## 工作边界

- 先读取代码、配置、现有文档、测试结果和 Git 历史，再形成结论
- 只执行用户授权的修改；未授权修改代码或测试时，只整理证据、更新文档并记录缺口
- 事实连接到仓库文件、提交、工具输出或用户明确提供的信息
- 信息不足时使用“待确认”，写明缺少的证据、责任人和期望时间

## 输出

- 优先更新项目已有文档，其次遵循项目目录约定，没有约定时写入 `docs/aspice/`
- 使用相对链接连接代码、需求、测试、问题、变更和图形文件
- 保留仍然有效的人工内容，只更新冲突、重复或失效部分

## 首屏摘要

开头固定保留五项：

- **状态**：未开始、进行中、受阻或完成
- **结论**：当前最重要的判断
- **关键决策**：已经生效并影响后续工作的决定
- **主要风险**：最多三项
- **待确认**：缺失信息、责任人和期望时间；没有时写“无”

## 正文

- 首屏摘要后最多保留四个二级章节
- 每个需求、组件、验证项、问题或变更只在一张主表中完整描述，其他位置使用 ID 引用
- 展开失败、冲突、风险和缺口，符合预期的项目保留在主表中
- 除表格数据行和图形外，正文不超过 60 行
- 表格不超过五列
- 复杂关系使用当前 Skill 指定的图形格式；没有指定时使用一张小型 Mermaid 图
- 原始日志和详细结果使用链接引用
- 标题、列表项和短句说明结尾不加句号
- 默认使用简体中文，代码标识符、接口名、单位和 ASPICE 过程 ID 保持原文

## 交付检查

- 首屏五项均有真实内容
- 所有关键事实都有证据或标记为待确认
- 所有受影响工作产品均已纳入追溯或缺口清单
- 每个事实只有一个完整描述位置
- 文档没有空章节、占位文本和标准复述
- 文档路径、相对链接和命令可直接使用
