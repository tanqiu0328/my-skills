# 项目管理

## 执行步骤

1. 完整读取 [ASPICE 文档契约](references/document-contract.md)、[过程参考](references/process.md) 和 [文档模板](assets/template.md)
   - MAN.3 使用模板中的工程进度首屏替代通用摘要字段，其他证据和真实性规则继续适用
   - 完成标准：已明确工程计划与审核证据的层次、状态口径和篇幅边界
2. 识别目标和真实进度
   - 优先读取 PRD、交付范围、仓库状态、代码、测试、Issue、实际人员和关键依赖；合同和组织流程存在时再纳入
   - 将“代码实现完成”“目标构建完成”“样板或 APP 验收完成”分开判断
   - 完成标准：已完成、当前工作、下一步及真实协作条件均有证据，不沿用工具或 Issue 的模糊状态
3. 创建十秒进度摘要
   - 首屏依次写“项目状态、已经完成、当前工作、接下来、需要协作”五项
   - 使用工程名称表达事项，编号只在名称之后用括号补充
   - 完成标准：工程师无需查表即可在十秒内回答做到哪、现在做什么、后面做什么和谁需要参与
4. 建立单一工程推进表
   - 按“已完成、现在、下一步、然后、最后、可选”表达实际顺序，将里程碑和工作包合并为可执行阶段
   - 当前和下一项写到可执行深度，远期只保留完成判断；详细任务使用 Issue 链接
   - 完成标准：每项工作的范围、完成判断、参与人和状态只在该表完整描述一次
5. 记录协作、风险和阶段门槛
   - 说明哪些工作可由当前工程师继续推进，哪些活动需要硬件、测试、客户或现场条件
   - 缺少姓名或精确时间时记录计划行动，只有无法开展对应活动时才局部受阻
   - 完成标准：每个风险或门槛都有实际影响和下一动作，未扩大为整个项目受阻
6. 后置 MAN.3 证据
   - 在最后一节汇总范围与生命周期、可行性与估算、工作包依赖、技能接口、监控偏差及完成条件
   - 详细 Issue、提交和测试只提供链接，审核编号和状态不重复回流到人话进度区
   - 完成标准：MAN.3 过程成果可追溯，工程推进区仍能独立阅读
7. 创建或更新 `project-management.md` 并执行交付检查
   - 使用中文行动状态，将工具状态转换为“待开始、进行中、等待协作、阶段门槛或完成”等工程语言
   - 完成标准：首屏十秒可读、同一状态只有一个完整描述位置、表格数据行默认不超过十五行，计划足以支持下一步决策
