# 软件详细设计与单元构建

## 执行步骤

1. 完整读取 [ASPICE 文档契约](references/document-contract.md)、[过程参考](references/process.md) 和 [文档模板](assets/template.md)
   - 完成标准：已明确详细设计基线、实现状态和缺口分类
2. 识别设计范围和现有实现
   - 优先读取软件架构、分配需求、组件接口和范围内代码；编码规则与单元测试存在时用于补充约束和证据
   - 完成标准：范围内组件、关键单元和当前实现状态已确认，缺口已局部化
3. 定义必要的静态设计
   - 为有独立职责或风险的单元说明 API、数据约束、依赖、资源所有权和错误契约
   - 简单适配或胶水代码使用单元主表和代码链接即可，不重复实现细节
   - 完成标准：每个关键单元的边界与接口可供实现和验证
4. 定义动态设计
   - 只展开状态机、关键算法、硬件交互、并发、时序及会影响结果的错误路径
   - 完成标准：关键正常路径与高风险失败路径均有确定行为，简单顺序逻辑可从主表和代码定位
5. 连接或构建软件单元
   - 只有用户授权实现时才修改代码；否则记录实现差距
   - 遵循项目已有编码规范和工具链
   - 完成标准：设计与现有或新增代码一致，未实现部分作为实现行动项而非设计基线阻塞项
6. 建立软件需求、软件架构、详细设计和代码的双向追溯并检查一致性
   - 完成标准：每个关键单元有上游依据，范围内代码的额外行为有明确来源或处置
7. 创建或更新 `software-detailed-design.md`，执行交付检查并报告路径、实现状态和缺口
   - 正文描述设计意图和约束，使用链接连接代码，避免逐行复述实现
   - 完成标准：详细设计可直接支持实现和 SWE.4，设计基线与实现进度没有混淆
