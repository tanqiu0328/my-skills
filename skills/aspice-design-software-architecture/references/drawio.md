# Draw.io 架构图规则

## 交付物

- 将 `software-architecture.drawio` 与 `software-architecture.md` 放在同一目录
- 先复制 [模板](../assets/software-architecture.drawio) 到目标目录，再修改未压缩的 XML 单元；保持稳定 ID，避免从空文件重建整个 `mxfile`
- 保留“静态架构”和“动态架构”两个页面
- Markdown 的“架构图”章节使用相对链接指向 `.drawio` 文件

## 视图选择

- 先检查项目已有架构图，在语义仍然正确时复用其边界、命名、颜色和阅读方向
- ASPICE 不指定 C4、UML、分层图或流程图等记法，选择能让当前工程团队最快读懂职责与运行关系的形式
- 嵌入式项目的静态页默认采用组件视图，动态页默认采用运行时与部署视图
- 实际源文件或 `app_*` 模块依赖属于实现视图，只有用户明确要求或它们本身就是架构组件时才作为静态页主体

## 静态架构页

- 展示部署节点、应用软件边界、外部对象、软件组件、硬件或平台服务和有方向的依赖
- 组件标签包含稳定 ID 与名称，例如 `ARCH-CMP-001\n温度采集`
- 连接线标签使用接口、数据或服务名称
- 需求分配和详细属性保留在 Markdown 主表，图中只呈现理解关系所需的信息

## 动态架构页

- 嵌入式项目优先展示部署节点、任务、ISR、时钟、队列、快照、同步调用和硬件访问所有权
- 跨任务命令与状态流标明是队列、最新值、共享快照或同步调用；容量和周期未知时连接验证项，不编造数值
- 一个执行上下文内可嵌套相关逻辑组件，明确唯一写入者和顺序执行关系
- 没有并发或部署关系时，可改用一至四个关键场景；失败路径只保留影响架构决策的分支

## 视觉约定

- 已确认软件组件使用蓝色 `#dae8fc`
- 待确认或提议内容使用黄色 `#fff2cc`
- 外部系统、硬件和平台使用灰色 `#f5f5f5`
- 失败或高风险路径使用红色 `#f8cecc`
- 页面采用从左到右布局，每页控制在 15 个节点以内；更多内容使用容器分组
- 使用嵌套边界表达部署节点、应用软件和任务上下文，不用颜色代替边界语义
- 使用直角连接线并减少交叉，文字保持水平且不截断

## 验证

1. 使用 XML 解析器确认文件结构有效
2. 确认存在且仅存在“静态架构”和“动态架构”两个默认页面；用户明确要求增加视图时才新增页面
3. 确认所有连接线都具有 `source` 和 `target`
4. 确认已替换模板中的原样占位文本
   - 静态页检查“部署节点”“应用软件边界”“外部系统”“组件 A”“组件 B”“硬件适配”“目标硬件”和“硬件或平台服务”
   - 动态页检查“SDK / 网络任务上下文”“外部服务适配”“命令队列”“最新状态快照”“应用任务上下文”“组件 A”“组件 B”“硬件适配”“单调时钟”“目标硬件”和“RTOS / SDK / 硬件平台”
   - 发现任一未结合项目语义改写的占位内容即视为未完成
5. 确认 Markdown 链接可以解析到 `.drawio` 文件
6. 检测到 draw.io CLI 时导出 PNG 并检查节点重叠、连线交叉、文字截断和阅读顺序
