# Draw.io 架构图规则

## 交付物

- 将 `software-architecture.drawio` 与 `software-architecture.md` 放在同一目录
- 先复制 [模板](../assets/software-architecture.drawio) 到目标目录，再修改未压缩的 XML 单元；保持稳定 ID，避免从空文件重建整个 `mxfile`
- 保留“静态架构”和“动态架构”两个页面
- Markdown 的“架构图”章节使用相对链接指向 `.drawio` 文件

## 静态架构页

- 展示系统边界、外部对象、软件组件、硬件或平台服务和有方向的依赖
- 组件标签包含稳定 ID 与名称，例如 `ARCH-CMP-001\n温度采集`
- 连接线标签使用接口、数据或服务名称
- 需求分配和详细属性保留在 Markdown 主表，图中只呈现理解关系所需的信息

## 动态架构页

- 展示一至四个关键运行场景；简单项目一个端到端场景即可，使用从左到右的编号箭头表达先后关系
- 只标明实际存在且影响时序或并发的任务、中断、线程和模式边界
- 失败路径只保留影响架构决策的分支

## 视觉约定

- 已确认软件组件使用蓝色 `#dae8fc`
- 待确认或提议内容使用黄色 `#fff2cc`
- 外部系统、硬件和平台使用灰色 `#f5f5f5`
- 失败或高风险路径使用红色 `#f8cecc`
- 页面采用从左到右布局，每页控制在 15 个节点以内；更多内容使用容器分组
- 使用直角连接线并减少交叉，文字保持水平且不截断

## 验证

1. 使用 XML 解析器确认文件结构有效
2. 确认存在且仅存在“静态架构”和“动态架构”两个默认页面；用户明确要求增加视图时才新增页面
3. 确认所有连接线都具有 `source` 和 `target`
4. 确认已替换模板中的“外部系统”“组件 A”“组件 B”“硬件或平台”“触发事件”和“输出或执行器”
   - 同时搜索并清除“交互 1”“交互 2”“场景说明”等模板文本，发现任一占位内容即视为未完成
5. 确认 Markdown 链接可以解析到 `.drawio` 文件
6. 检测到 draw.io CLI 时导出 PNG 并检查节点重叠、连线交叉、文字截断和阅读顺序
