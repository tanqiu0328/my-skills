# SWE.2 软件架构设计

来源：Automotive SPICE PAM 4.0 中文版，第 52～53 页

## 目的

建立与软件需求一致、经过分析且包含静态和动态方面的软件架构

## 必须覆盖的成果

- 定义组件、外部接口、组件接口和依赖等静态方面
- 定义模式、行为、交互、时序和并发等动态方面
- 分析技术设计并记录关键设计依据
- 保持软件架构与软件需求的一致性和双向追溯
- 与受影响方约定并沟通软件架构

## 分析重点

- 功能分配与接口边界
- ROM、RAM、CPU、存储和时序
- 中断、抢占、任务、线程和模式切换
- 复用、平台适用性、自研或外购
- 原型、仿真或定性分析形成的证据

## 小团队适配

- 软件需求基线、硬件原理图和平台资料可以直接组成架构输入，不要求先补齐独立系统架构或正式 HSI 文件
- ASPICE 要求覆盖静态和动态方面，但不规定图形记法；项目已有的清晰视图、命名和颜色优先复用
- 组件数量按职责边界决定，简单 Demo 优先形成少量深组件
- 嵌入式静态架构优先使用组件视图；代码目录、源文件和实现模块依赖通常在 SWE.3 细化
- 嵌入式动态架构优先使用运行时与部署视图，明确任务、ISR、时钟、队列、快照、资源所有权和硬件访问上下文
- 业务场景流程用于补充关键交互，不在已有运行时关系时重复铺满整个动态页面
- ROM、RAM、CPU 和时序只在存在目标、风险或平台限制时量化；其他情况记录估算方法和后续测量入口
- 未完成的样板测量或原型验证作为验证项，不阻塞已经明确的组件、接口和关键动态行为
- 组件主表已经记录需求分配时，它就是双向追溯的唯一完整描述位置，不再复制逐需求矩阵
- 接口细节、状态转换和资源分析优先合并进组件、关键场景、决策与验证三张主表，详细设计阶段再展开实现级内容
- 架构评审可由项目负责人和实际开发者完成，同一人承担多个角色时记录一次真实决策
