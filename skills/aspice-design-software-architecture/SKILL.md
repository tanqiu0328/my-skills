# 软件架构设计

## 执行步骤

1. 完整读取 [ASPICE 文档契约](references/document-contract.md)、[过程参考](references/process.md)、[Draw.io 规则](references/drawio.md) 和 [文档模板](assets/template.md)
   - 完成标准：已明确架构基线、缺口分类和 Draw.io 交付规则
2. 识别架构范围和可用输入
   - 优先读取软件需求基线、硬件资料、平台约束和已有组件；系统架构、正式 HSI 和历史决策存在时再纳入
   - 将缺失文档、待验证参数和真正阻止组件划分的事实分开
   - 完成标准：范围内需求、可用平台事实及影响架构的局部缺口均已识别
3. 定义静态架构
   - 描述外部接口、软件组件、职责、接口、依赖和需求分配
   - 同步更新 Draw.io 的“静态架构”页面
   - 完成标准：每项范围内软件需求已分配到组件或有明确的跨组件处理方式，组件与依赖在图中可见
4. 定义动态架构
   - 只展开影响组件协作的关键模式、状态、数据流、调用关系、中断、任务、时序和并发交互
   - 同步更新 Draw.io 的“动态架构”页面
   - 完成标准：每个关键场景能够从输入追踪到响应，实际存在的跨上下文交互有明确约束
5. 分析并约定架构
   - 分析当前范围适用的功能、ROM、RAM、CPU、时序、可靠性、复用及自研或外购决策
   - 上游没有数值目标时记录估算依据和验证项，不编造资源阈值
   - 完成标准：关键决策有依据，高风险或待量化约束有局部验证入口
6. 建立需求与架构的双向追溯并检查语义一致性
   - 默认以组件主表中的需求分配作为追溯源；已能检查完整覆盖时不再生成逐需求镜像矩阵，只列未分配或冲突项
   - 完成标准：需求分配完整，架构没有无法解释的功能或接口
7. 创建或更新 `software-architecture.md` 和同目录的 `software-architecture.drawio`
   - 新建图时从 [Draw.io 模板](assets/software-architecture.drawio) 开始，Markdown 使用相对链接指向图文件
   - 默认只保留模板中的组件、关键场景、决策与验证三张主表；只有关系无法读清时才增加一张必要明细表
   - 校验 XML、页面、节点、连接线和模板占位内容；检测到 draw.io CLI 时导出 PNG 预览并检查布局
   - 完成标准：架构可作为 SWE.3 输入，图文件可由 diagrams.net 编辑，包含静态和动态两个页面且与正文一致
