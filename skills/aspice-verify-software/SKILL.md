# 软件验证

## 执行步骤

1. 完整读取 [ASPICE 文档契约](references/document-contract.md)、[过程参考](references/process.md) 和 [文档模板](assets/template.md)
   - 完成标准：已明确验证计划、执行结果和发布结论的边界
2. 确定验证分支和发布范围
   - 区分制定验证计划、选择回归集、执行验证或汇总发布结论，并读取相应的软件需求、软件基线和已有证据
   - 完成标准：发布范围内的软件需求、软件版本和本次交付状态已确定
3. 定义软件验证措施
   - 为功能和非功能需求定义技术、顺序、环境、通过/失败以及准入/准出准则
   - 完成标准：每项范围内软件需求至少有一项足以证明符合性的验证措施
4. 根据发布范围、优先级、变更影响和回归准则选择本次执行集
   - 完成标准：所有受变更影响和高风险需求均已选择，排除项有明确理由
5. 执行或收集集成软件验证结果
   - 用户只要求计划时将措施标记为“已规划”，不把尚未执行写成阻塞
   - 用户授权执行时保存软件版本、环境、命令、数据和通过/失败状态
   - 完成标准：执行分支中的每项已选措施都有结果或局部受阻原因，异常连接到问题记录
6. 建立软件需求、验证措施与结果的双向追溯，评估偏差和发布结论
   - 发布结论只使用已执行证据；计划分支只报告覆盖计划，不宣称软件通过验证
   - 完成标准：每项范围内需求都有计划或结果状态，失败和发布阻塞项有处置
7. 创建或更新 `software-verification.md`，执行交付检查并报告路径、覆盖率、偏差和结论
   - 完成标准：文档明确是验证计划、执行中记录或完成报告；只有完成报告支持发布判断
