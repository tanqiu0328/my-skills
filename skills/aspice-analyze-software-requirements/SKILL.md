# 软件需求分析

## 执行步骤

1. 完整读取 [ASPICE 文档契约](references/document-contract.md)、[过程参考](references/process.md) 和 [文档模板](assets/template.md)
   - 完成标准：已明确基线判定、证据规则和输出结构
2. 识别交付目标、发布范围和可用输入
   - 优先读取已有 PRD、利益相关方需求、硬件资料和软件需求；代码、配置、测试与 Git 只在能补充当前需求事实时取证
   - 软件主导项目可直接使用利益相关方需求和给定运行环境，独立系统需求、系统架构或 HSI 文档缺失不自动形成阻塞项
   - 完成标准：范围、可用输入和会改变软件行为或验收的证据缺口均已识别
3. 建立可开发的软件需求基线
   - 先转换明确的上游需求，再补充当前范围确实需要的接口、时序、资源、安全或运行环境约束
   - 为需求分配 ID、来源、优先级、验收条件和验证方式
   - 将未规定且可逆的细节定为简明工程默认，将不确定性分为需求阻塞项、验证项和阶段门槛
   - 完成标准：每项范围内需求可验证，每项工程默认有依据，未引入与交付目标无关的需求
4. 判定并局部化缺口
   - 仅当缺少的事实阻止相关需求实现、客观验收或安全开展下一活动时，将其标记为阻塞项
   - 将原型验证、板级核对和后续阶段准入分别记录到对应需求或阶段，不扩大为整份文档受阻
   - 完成标准：每个缺口都有类型、影响范围和关闭时点，非阻塞事项未被写成基线前置条件
5. 建立双向追溯并检查一致性
   - 连接软件需求与可用的系统需求、利益相关方需求、系统架构或给定运行环境
   - 完成标准：每项软件需求有上游来源，每项适用上游需求有软件去向或明确的非软件去向
6. 创建或更新 `software-requirements.md`
   - 优先更新已有文档，否则写入 `docs/aspice/`
   - 使用模板的首屏摘要，正文只保留输入与范围、需求、追溯、验证项和阶段门槛
   - 完成标准：文档能够直接支持设计、估算和验证，且基线状态不受后续阶段事项误判影响
7. 执行交付检查并向用户报告文档路径、关键结论和阶段门槛
   - 完成标准：契约中的每项交付检查均通过
