# 软件需求分析

## 执行步骤

1. 完整读取 [ASPICE 文档契约](references/document-contract.md)、[过程参考](references/process.md) 和 [文档模板](assets/template.md)
   - 完成标准：已明确输出位置、证据规则和 SWE.1 全部过程成果
2. 检查利益相关方需求、系统需求、系统架构、HSI、现有软件需求、问题与变更记录
   - 优先使用 `rg`、项目构建配置、接口定义和 Git 历史取证
   - 完成标准：每个输入来源均已检查，缺失输入已列入待确认项
3. 建立软件需求集合
   - 定义功能、性能、时序、资源、接口、诊断、安全和运行环境约束
   - 为需求分配 ID、来源、优先级、验收条件和验证方式
   - 分析正确性、无歧义性、可行性、相互依赖及对运行环境的影响
   - 完成标准：每项范围内需求可验证且不存在未说明的冲突
4. 建立双向追溯并检查一致性
   - 连接软件需求与系统需求、系统架构或给定运行环境
   - 完成标准：每项软件需求有上游来源，每项适用上游需求有软件侧去向或明确理由
5. 创建或更新 `software-requirements.md`
   - 优先更新已有文档，否则写入 `docs/aspice/`
   - 使用模板的首屏摘要，正文只保留需求、追溯、影响和未决项
   - 完成标准：受影响方能够据此设计、估算和验证软件
6. 执行交付检查并向用户报告文档路径、关键结论和缺口
   - 完成标准：契约中的每项交付检查均通过
