# SWE.1 软件需求分析

来源：Automotive SPICE PAM 4.0 中文版，第 50～51 页

## 目的

建立一组经过结构化和分析、与适用上游需求、架构及给定运行环境一致的软件需求

## 必须覆盖的成果

- 定义功能性和非功能性软件需求
- 结构化需求并确定优先级
- 分析正确性、技术可行性和相互依赖
- 分析需求对运行环境的影响
- 保持软件需求与适用的系统需求、系统架构或利益相关方需求之间的一致性和双向追溯
- 与受影响方约定并沟通软件需求和环境影响

## 小团队适配

- PAM 规定过程成果和信息项，不规定必须拆成多少份独立文档
- 只有软件开发时，利益相关方需求和给定运行环境可直接作为软件需求输入与追溯对象
- HSI 已存在时将其作为输入；缺少独立 HSI 文件不等于缺少接口事实
- 约定和沟通可由评审记录、问题记录、提交记录或用户确认证明，形式与项目规模相称
- 功能、性能、资源、诊断和安全等类别按当前交付范围选用，不机械补齐全部类别

## 信息分类

| 类型 | 判定 | 基线处理 |
|---|---|---|
| 软件需求 | 规定可观察行为、接口或必须满足的约束 | 写入需求主表并验证 |
| 工程默认 | 上游未规定且团队可在授权范围内决定的可逆细节 | 直接确定并标记依据 |
| 验证项 | 通过原型、目标 APP、样板或测试确认可行性和呈现 | 关联受影响需求，不阻塞无关开发 |
| 阶段门槛 | 开展联调、上电、发布或验收前必须满足的外部条件 | 只阻塞对应活动 |
| 需求阻塞项 | 缺少事实导致相关行为无法定义、实现或客观验收 | 局部标记受阻并请求决策 |

## 基线判定

当范围内需求可验证、上游覆盖完整且软件需求阻塞项已关闭时，可形成开发基线

验证项和后续阶段门槛可以随基线存在；只有核心范围、关键行为或验收口径未确定时，整份文档才标记为受阻

## 证据重点

- 需求及其属性
- 正确性、可行性和运行环境影响分析
- 上游需求、架构与软件需求之间的一致性证据
- 受影响方可获得约定内容的证据

追溯链接只证明关系存在，还要检查上下游含义是否一致
