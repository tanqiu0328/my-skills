# ASPICE 文档契约

## 工作边界

- 先读取与当前范围直接相关的现有文档，再按需检查代码、配置、测试结果和 Git 历史
- 只执行用户授权的修改；未授权修改代码或测试时，只整理证据、更新文档并记录缺口
- 事实连接到仓库文件、提交、工具输出或用户明确提供的信息
- 将缺失文档与缺失事实分开；只有会改变行为、接口、验收或安全准入的缺失事实进入缺口清单
- 对上游未规定且团队有权决定的可逆细节采用工程默认，并记录依据和验证方式

## 输出

- 优先更新项目已有文档，其次遵循项目目录约定，没有约定时写入 `docs/aspice/`
- 使用相对链接连接代码、需求、测试、问题、变更和图形文件
- 保留仍然有效的人工内容，只更新冲突、重复或失效部分

## 首屏摘要

开头固定保留五项：

- **状态**：草案、可开发基线、受阻或完成
- **结论**：当前能否进入后续开发及结论边界
- **关键决策**：已经生效并影响后续工作的决定
- **主要风险**：最多三项
- **待确认**：只列软件需求阻塞项；没有时写“无软件需求基线阻塞项”并引用验证项或阶段门槛

## 正文

- 首屏摘要后最多保留四个二级章节
- 每个需求、组件、验证项、问题或变更只在一张主表中完整描述，其他位置使用 ID 引用
- 需求主表之外只展开追溯覆盖、验证项和阶段门槛
- 除表格数据行和图形外，正文不超过 60 行
- 表格不超过五列
- 表格足以表达时省略图形，复杂关系才使用一张小型 Mermaid 图
- 原始日志和详细结果使用链接引用
- 标题、列表项和短句说明结尾不加句号
- 默认使用简体中文，代码标识符、接口名、单位和 ASPICE 过程 ID 保持原文

## 交付检查

- 首屏五项均有真实内容
- 所有关键事实都有证据、工程默认或明确的缺口类型
- 所有范围内上游需求均有软件去向或非软件去向
- 软件主导项目未因缺少独立上游文档而被误判为受阻
- 验证项和阶段门槛未被误写成整份需求基线阻塞项
- 每个事实只有一个完整描述位置
- 文档没有空章节、占位文本和标准复述
- 文档路径、相对链接和命令可直接使用
