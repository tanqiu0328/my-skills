# ASPICE 文档契约

## 工作边界

- 先读取与当前任务直接相关的现有文档，再按需检查代码、配置、测试结果和 Git 历史
- 只执行用户授权的修改；未授权修改代码或测试时，只整理证据、更新文档并记录后续行动
- 事实连接到仓库文件、提交、工具输出或用户明确提供的信息
- 将缺少独立文档与缺少工程事实分开，只有影响当前工作产品使用的缺失事实才形成阻塞项
- 按项目实际记录角色，同一人可以承担多个角色，不虚构审批人、团队或沟通记录

## 输出

- 优先更新项目已有文档，其次遵循项目目录约定，没有约定时写入 `docs/aspice/`
- 使用相对链接连接代码、需求、测试、问题、变更和图形文件
- 保留仍然有效的人工内容，只更新冲突、重复或失效部分

## 首屏摘要

开头固定保留五项：

- **状态**：草案、已规划、进行中、可用基线、受阻或完成
- **结论**：当前工作产品能支持什么活动及其边界
- **关键决策**：已经生效并影响后续工作的决定
- **主要风险**：最多三项
- **待确认**：只列当前交付阻塞项；没有时写“无当前交付阻塞项”并引用验证项或阶段门槛

## 正文

- 首屏摘要后最多保留四个二级章节
- 每个组件、单元、验证项、问题、变更或工作包只在一张主表中完整描述，其他位置使用 ID 引用
- 同一追溯关系只完整记录一次；主表已能检查覆盖时不再生成正向、反向两套镜像矩阵
- 明确区分计划、执行结果和发布结论，状态不超出已有证据
- 展开失败、冲突、风险和缺口，正常项保留在主表中
- 除表格数据行和图形外，正文不超过 60 行
- 表格不超过五列
- 表格足以表达时省略图形；当前 Skill 指定图形格式时遵循该格式
- 原始日志和详细结果使用链接引用
- 标题、列表项和短句说明结尾不加句号
- 默认使用简体中文，代码标识符、接口名、单位和 ASPICE 过程 ID 保持原文

## 缺口分类

| 类型 | 判定 | 处理 |
|---|---|---|
| 当前阻塞项 | 缺少事实导致当前工作产品无法设计、执行、验收或安全使用 | 局部标记受阻并请求决策 |
| 验证项 | 需要原型、样板、工具或测试补充证据 | 关联对象和执行阶段 |
| 阶段门槛 | 后续联调、上电、发布或关闭前必须满足 | 只阻塞对应活动 |
| 改进项 | 提升成熟度但不影响当前范围 | 记录优先级，不影响当前基线 |

## 交付检查

- 首屏五项均有真实内容
- 范围、计划、结果和结论互相一致
- 关键事实都有证据、工程决策或明确的缺口类型
- 缺少独立过程文档未被误判为缺少工程事实
- 阻塞影响已局部化，验证项和阶段门槛未扩大为整份文档受阻
- 每个事实只有一个完整描述位置
- 文档没有空章节、占位文本和标准复述
- 文档路径、相对链接和命令可直接使用
