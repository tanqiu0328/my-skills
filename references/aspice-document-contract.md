# ASPICE 文档契约

## 工作边界

- 先读取项目中的代码、配置、现有文档、测试结果和 Git 历史，再形成结论
- 只执行用户授权的修改；未授权修改代码或测试时，仅整理证据、更新文档并记录缺口
- 事实必须能链接到仓库文件、提交、工具输出或用户明确提供的信息
- 信息不足时使用“待确认”，同时写明缺少什么证据以及由谁确认

## 输出位置

1. 优先更新项目已有的同类文档
2. 其次遵循项目现有文档目录和命名规则
3. 没有既有约定时写入 `docs/aspice/`
4. 使用相对链接连接代码、需求、测试报告、问题和变更记录

更新文档时保留仍然有效的人工内容，只重写与当前证据冲突、重复或失效的部分

## 首屏摘要

每份文档开头保留以下五项，一项一个短句或短列表：

- **状态**：未开始、进行中、受阻或完成
- **结论**：当前最重要的判断
- **关键决策**：已经生效并影响后续工作的决定
- **主要风险**：最多三项
- **待确认**：缺失信息、责任人和期望时间；没有时写“无”

读者应在首屏内判断当前状态、能否继续以及需要谁采取行动

## 正文规则

- 面向工程师和评审者写作，直接呈现项目事实、决策和证据
- 模板定义默认章节结构，正文在首屏摘要后最多保留四个二级章节
- 每个需求、组件、验证项、问题或变更只在一张主表中完整描述，其他位置使用 ID 引用
- 采用例外式写法：正文展开失败、冲突、风险和缺口，符合预期的项目留在主表中
- 除表格数据行和 Mermaid 图外，默认正文不超过 60 行
- 重复型结构使用表格，表格不超过五列
- 三个以上对象存在依赖、分支或时序关系时，使用一张小型 Mermaid 图
- 详细日志和原始结果使用链接引用，正文保留摘要
- 附录只在用户明确要求或主表无法承载关键安全信息时创建
- 标题、列表项和短句说明结尾不加句号
- 默认使用简体中文；代码标识符、接口名、单位和 ASPICE 过程 ID 保持原文
- PAM 过程说明只放在 Skill 的 `references/process.md` 中，不写入交付文档

## 追溯与状态

- 使用项目已有 ID；缺少 ID 时分别采用 `SWR-###`、`ARCH-###`、`UNIT-###`、`VER-###`、`PR-###`、`CR-###`
- 追溯关系至少包含来源、目标和当前状态
- 链接存在不等于内容一致；同时检查上下游语义、约束和状态是否一致
- 状态为失败、受阻或部分完成时，连接对应问题记录或待执行行动

## 交付检查

交付前确认：

- 首屏五项均有真实内容
- 首屏摘要后最多有四个二级章节
- 除表格数据行和 Mermaid 图外，正文不超过 60 行
- 每个事实只有一个完整描述位置，其余位置使用 ID 或链接引用
- 所有关键事实都有证据或明确标记为待确认
- 所有受影响工作产品均已纳入追溯或缺口清单
- 没有空章节、占位文本、重复说明和面向审计者的标准复述
- 文档路径、相对链接和命令可由项目成员直接使用
